Agent connections & models

        
Connect Claude Code, Codex, and any local/self-hosted models. These apply across every repository.

      

      
    

    

      

        
CLI agents

        
Claude Code and Codex run as local CLIs. A green dot means the binary was found on this machine. Set a model to pin every dispatch to it, or leave it blank to use the CLI's own default.

        

          Setup guide: Claude Code & Codex
          

            

              
Claude Code

              
    
                
  1. Install the CLI and sign in once: run claude in a terminal and complete the login flow. This dashboard shells out to that same install (claude -p ...), so it needs to already be authenticated before you dispatch from here.
    2. 
                
  2. On Windows the install location is auto-detected. If the status dot below stays red, set AGENT_MEMORY_SYNC_CLAUDE_BIN to the full path of claude.exe before starting the dashboard.
    3. 
                
  3. To have Claude Code sessions recorded into the shared timeline (not just dispatched from here), merge config/generated/claude_settings.json into %USERPROFILE%\.claude\settings.json, then restart Claude Code.
    4. 
                
  4. Leave the model field below blank to use Claude Code's own default, or type an exact id (e.g. claude-sonnet-5) to pin every future dispatch to it.
    5. 
              

            

            

              
Codex

              
    
                
  1. Install the CLI and run codex login once. Dispatch shells out to codex exec headlessly, so it needs that same authenticated session.
    2. 
                
  2. If the status dot below stays red on Windows, set AGENT_MEMORY_SYNC_CODEX_BIN to the full path of codex.exe.
    3. 
                
  3. Codex ships in two flavors, check which one you have before wiring up history capture: open %USERPROFILE%\.codex\config.toml. Already has one or more [mcp_servers.*] entries? You're on the MCP-capable build, merge config/generated/codex_mcp.toml. See a /hooks command or a hooks.json instead? Merge config/generated/codex_hooks.toml. Restart Codex after merging either one.
    4. 
                
  4. The model field below works the same as Claude Code's: blank uses Codex's own default, or pin an id like gpt-5-codex.
    5. 
              

            

          

        

        

      

      

        
Local & self-hosted models

        
Any endpoint that speaks the OpenAI chat-completions API can plug in as a third dispatch lane, read-only for now.

        

          Setup guide: local & self-hosted models
          

            

              This lane talks OpenAI-compatible HTTP: dispatch calls POST {base_url}/v1/chat/completions
              and the Test button calls GET {base_url}/v1/models. There's no tool
              loop yet, so a local model can read the pooled context and answer, but it can't edit files, unlike
              Claude Code and Codex. Always deploy it with "allow edits" off.
            

            

              
Ollama

              
    
                
  1. Install Ollama, then pull a model: ollama pull gemma3:4b.
    2. 
                
  2. Run ollama serve (or just launch the app) — it exposes an OpenAI-compatible API on port 11434 by default.
    3. 
                
  3. Add it below with base URL http://localhost:11434 and model gemma3:4b (or whichever tag you pulled). No API key needed.
    4. 
              

            

            

              
LM Studio

              
    
                
  1. Load a model in LM Studio, then open the Developer tab and start the local server.
    2. 
                
  2. Default base URL is http://localhost:1234. Use the model id shown as loaded (e.g. lmstudio-community/gemma-2-9b).
    3. 
              

            

            

              
vLLM

              
    
                
  1. vllm serve <hf-model-id> --port 8000 starts an OpenAI-compatible server.
    2. 
                
  2. Base URL is http://localhost:8000; model is the same id you passed to vllm serve.
    3. 
              

            

            

              
LocalAI & text-generation-webui

              
    
                
  1. LocalAI: run its Docker image; it defaults to port 8080 with an OpenAI-compatible API built in.
    2. 
                
  2. text-generation-webui: launch with --api; its OpenAI-compatible extension defaults to port 5000.
    3. 
              

            

            

              
Hosted OpenAI-compatible APIs (Together, Groq, OpenRouter, ...)

              
    
                
  1. Set an environment variable with the provider's API key in the environment the dashboard runs in, e.g. OPENROUTER_API_KEY.
    2. 
                
  2. Use that provider's base URL and one of its model ids, then put the env var's name — not the key itself — in the "API key env var" field below. It's read from the dashboard process's environment at request time and never stored.
    3. 
              

            

            

              Field reference: id is an internal slug (unique, letters/numbers/dashes); display name is what shows in the dispatch picker; base URL has no trailing /v1. Click Test after adding to confirm the endpoint answers.
