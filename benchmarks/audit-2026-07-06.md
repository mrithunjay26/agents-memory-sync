# AgentMemorySync audit and scale benchmark

Date: 2026-07-06

## Executive conclusion

AgentMemorySync is already useful as a local, cross-agent memory and
orchestration layer. Its strongest measured advantage is reducing the amount
of retained evidence sent to an agent for a focused lookup or briefing. It is
not yet defensible to claim measured end-to-end token savings or Bluebird
parity.

The current working tree passes all 106 tests, Python compilation, JavaScript
syntax checks, dependency consistency checks, and `git diff --check`.

Bluebird could not be run: this environment has no Bluebird MCP tool, endpoint,
credentials, configuration, dataset adapter, or public product specification.
The official public Microsoft MCP catalog does not list a Bluebird server.
AgentMemorySync's benchmark therefore continues to mark Bluebird as not run.
A numerical head-to-head result would require Bluebird predictions over the
same exported corpus and queries.

## Measured retrieval regression

The built-in 180-query synthetic suite produced:

| System label | Recall@5 | nDCG@10 | MRR | Mean evidence tokens | p50 | p95 |
|---|---:|---:|---:|---:|---:|---:|
| Legacy `LIKE` | 1.000 | 1.000 | 1.000 | 145.3 | 3.06 ms | 3.32 ms |
| Production search | 0.994 | 0.996 | 0.995 | 162.6 | 63.27 ms | 65.01 ms |

These numbers are regression evidence, not product parity evidence. Each query
shares a distinctive scenario slug, symbol, path, or marker with its relevant
document. That makes the suite intentionally easy and explains the perfect
legacy baseline.

The report currently calls the production row `fts5`, but the code invokes
`store.search_shared_context`, which is the FTS5 plus hashed-ngram hybrid path.
The report also says dense, hybrid, and code graph are unavailable even though
hashed vectors and a Python code graph now exist. Those labels are stale.

## Large-repository lab

Synthetic Python repositories were processed through the production index,
search, and architecture paths. Every search includes the current automatic
repository refresh.

| Files | Source token estimate | Cold index | No-change refresh | Search p50 | DB size | Search response | Reduction vs whole source |
|---:|---:|---:|---:|---:|---:|---:|---:|
| 100 | 13,175 | 0.94 s | 0.14 s | 0.15 s | 1.09 MB | 250 tokens | 98.10% |
| 1,000 | 131,750 | 7.68 s | 0.54 s | 0.57 s | 8.78 MB | 250 tokens | 99.81% |
| 5,000 | 658,750 | 34.26 s | 2.32 s | 2.49 s | 42.79 MB | 250 tokens | 99.96% |

Exact-marker top-1 recall was 100% at all three sizes. This proves exact
localization and payload reduction for the generated corpus. It does not prove
semantic task completion. The 5,000-file latency is a real scaling concern:
`search_code`, `explain_symbol`, `find_references`, and `get_repository_map`
all call `_ensure_index`, which walks and hashes the repository before reading
the index.

The architecture response was capped near 20,156 estimated tokens at 5,000
files and marked truncated. Capping prevents unbounded model payloads, but a
20K-token map is still expensive for a preliminary navigation call.

## Large-history and context lab

The synthetic history corpus used 2,000 recorded native tokens per event, a
compact digest per event, and the default 15-entry active window.

| Events | Retained digest tokens | Active briefing | Briefing reduction | Bundle build | Dashboard bundle payload | Search p50 |
|---:|---:|---:|---:|---:|---:|---:|
| 100 | 4,700 | 1,048 | 77.70% | 4.34 ms | 22,133 tokens | 34.28 ms |
| 1,000 | 47,000 | 1,048 | 97.77% | 15.34 ms | 185,714 tokens | 83.43 ms |
| 10,000 | 470,000 | 1,048 | 99.78% | 97.95 ms | 1,823,719 tokens | 84.70 ms |

The agent-facing briefing scales well because the working set is bounded.
The dashboard/API representation does not: `get_context_bundle` reads and
returns every event before selecting the active 15. Pagination or a separate
briefing endpoint is required before very large histories are practical.

## Current-project token audit

Read-only inspection of this repository's current store found:

- 86 non-telemetry corpus entries, 78 included, 15 currently active.
- 1,845,776 estimated tokens of retained raw transcript text.
- 353,625 estimated raw transcript tokens represented by the current active
  15 entries.
- 1,515 estimated tokens in those entries' compact summaries, before the
  briefing header and search instructions.
- 43,187,406 recorded native session tokens across included entries and
  11,686,480 across the active 15 entries.
- 24 recorded context deliveries totaling 123,279 estimated delivered tokens,
  or about 5,137 per delivery.
- Delivery snapshots sum to 475,056,831 historical native tokens because the
  active sessions' original usage is counted again for each reuse.

The measured compression from the active entries' retained transcript text to
their compact summaries is about 99.57%. That is a useful payload-size result.
It is not proof that 99.57% of model tokens or cost were saved.

## Material audit findings

### High: `verified_tokens_saved` is not verified savings

Telemetry subtracts delivered briefing size from the native token cost of the
historical sessions represented by the active entries, then repeats that
subtraction for every delivery. Historical native usage includes iterative
prompt context, reasoning/output, cache traffic depending on provider, tool
work, and work unrelated to the retained digest. No control session measures
what the receiving agent would otherwise have consumed.

The current data would report a gap near 474.9 million tokens from only 24
deliveries. That number is cumulative reuse potential under a strong
counterfactual, not verified avoided usage. Rename it to `native_work_reuse_gap`
or remove it from savings UI. Reserve “verified savings” for paired tasks run
with and without AgentMemorySync using the same model, prompt, repository
revision, cache state, and success criteria.

README language is internally inconsistent: it says telemetry never converts
historical usage into claimed savings, then documents exactly that conversion.

### High: repository reads re-index synchronously

Every code search, symbol explanation, reference lookup, and architecture map
calls `_ensure_index`, which invokes a full discovery/hash refresh and Git
history scan. This produced a 2.49-second exact lookup at 5,000 files even
after a no-change index. Refresh should be explicit, backgrounded, scheduled,
or guarded by a cheap freshness check.

### Medium: benchmark labels no longer match production

The `fts5` row is actually hybrid. The `not_run` section says hybrid and code
graph do not exist. This makes the benchmark report misleading even though its
metric calculations are valid.

### Medium: context management API is unbounded

The model briefing is bounded, but `get_context_bundle` returns all entry
metadata. The lab's 10,000-event response was about 1.82 million estimated JSON
tokens. Split exact briefing delivery from paginated management/search APIs.

### Medium: semantic scope is narrower than the name suggests

The vector leg is a local 256-dimensional hashed word/bigram/character-ngram
representation over only the 300 most recent entries. It handles morphology,
reordering, and typos, but not learned conceptual similarity, and older entries
receive lexical retrieval only. Describe it as fuzzy hashed-ngram retrieval.

### Medium: enterprise controls stop at the dashboard boundary

Dashboard APIs enforce repository RBAC. The local MCP tools directly accept a
path and do not check tracked-project grants, and hooks write directly to the
store. That can be a valid local-stdio trust model, but it must not be presented
as end-to-end enterprise authorization.

The audit chain is hash-linked but neither keyed nor externally anchored. A
writer with database access can rewrite the data and recompute the chain. It is
tamper-evident against accidental or partial modification, not tamper-proof.

### Known operational limits confirmed

- Dispatch processes are tracked in in-memory dictionaries and do not survive
  dashboard restart.
- Coordinated editing agents share one working tree; there is no worktree or
  container isolation.
- Structural indexing is Python-only. Other supported languages are lexical.
- Secret detection is pattern-based and SQLite is not encrypted at rest.

## Total advantages

AgentMemorySync's defensible advantages are:

1. A provider-neutral episodic memory shared by Claude Code and Codex.
2. Human curation: pin, exclude, edit summaries, categorize, and add notes
   without changing the original audit evidence.
3. A small, identical active briefing plus searchable retained transcripts.
4. Immutable dispatch-time context snapshots and delivery receipts.
5. Local orchestration, monitoring, cancellation, redirection, handoff, and
   dashboard-mediated agent questions.
6. Parallel coordinated follow-ups with per-task agent/model/edit settings.
7. Local-first storage and dependency-free fuzzy retrieval with no embedding
   API key or hosted vector database.
8. Repository search, Python symbol/call/reference indexing, architecture maps,
   Git provenance, and agent file-touch history through one MCP surface.
9. Dashboard-scoped RBAC, secret redaction, retention controls, and audit
   exports for self-hosted teams.
10. Reproducible instrumentation that can distinguish retained corpus size,
    active briefing size, actual delivery size, and provider-native usage once
    the misleading savings label is corrected.

These advantages are complementary to a repository-understanding product.
AgentMemorySync's differentiator is cross-agent continuity and orchestration;
its current code intelligence should not be marketed as Bluebird-equivalent
without an executable common benchmark.

## Recommended next experiments

1. Run a paired 30-50 task A/B study on real repositories: same task/model and
   clean cache, with versus without AgentMemorySync. Measure successful
   completion, input/output/cache tokens, wall time, retries, and answer/change
   correctness.
2. Export the 180-query corpus and obtain Bluebird ranked document IDs. Score
   them through `benchmark.py --predictions bluebird=<file>`.
3. Add real repositories and human relevance labels; the current synthetic
   suite is too easy to rank retrieval quality.
4. Separate index refresh from read tools and rerun the 5,000-file lab.
5. Paginate context management and rerun the 10,000-event lab.
